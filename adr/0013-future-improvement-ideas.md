# ADR 0013: 改善アイデア集 — 「人生を振り返って未来をよくする」ための機能候補

## ステータス

Proposed（アイデア集。個別に着手する際は必要に応じて独立したADRに昇格させる）

## コンテキスト

umi.mikan は「記録する」（日記CRUD・オフライン対応iOSアプリ）と「探す」（キーワード検索・セマンティック検索）が既に強い。
一方でプロダクトのコンセプトである「ユーザの人生を振り返って未来をよくできる日記サービス」を実現するには、次の2つが不足している。

1. **振り返りが自然に起きる仕組み** — ユーザが能動的に検索しなくても過去が目に入る導線
2. **振り返りを未来の行動につなげる仕組み** — 過去の記録から気づきを得て、行動を変えるためのフィードバックループ

本ADRはこの2軸を埋める機能候補を優先度順にまとめたものである。

### 既存資産（再利用前提）

- 月次AIサマリー・トレンド分析・ハイライト生成（Scheduler + Redis Pub/Sub + Subscriber + Gemini API）
- セマンティック検索（pgvector halfvec 埋め込み、ADR 0009）
- エンティティ管理（entity + alias、ADR 0002）
- iOS: WidgetKit 基盤（Live Activity 実装済み）、オフラインストア、SyncManager
- 関連する Proposed ADR: 0010（自己分析レポート）、0011（人間関係の可視化）

## 機能候補（優先度順）

### 1. 「n年前の今日」（On This Day）

**分類**: 振り返り導線 / **実装コスト**: 小 / **LLM**: 不要

- **概要**: 過去の同月同日の日記を全年分まとめて表示する。ホーム画面のセクション、iOSウィジェット、朝の通知に載せる。
- **なぜ**: 振り返り系サービスの定番で費用対効果が最も高い。日記サービス最大の敵は「書かなくなること」であり、「毎日開く理由」を作ることが継続の鍵になる。
- **実装方針**:
  - 過去の同月同日を全年分取得する RPC を1本追加（`GetDiaryEntriesOnThisDay` など。日付指定の `GetDiaryEntries` の応用）
  - Web: ホームにセクション追加。iOS: WidgetKit ウィジェット + ローカル通知
  - LLM 不要なので全ユーザに提供可能

### 2. エンティティ（人物）タイムライン

**分類**: 振り返り導線 / **実装コスト**: 小〜中 / **LLM**: 一部利用

- **概要**: entity 詳細ページに「初登場日」「登場頻度の推移グラフ」「この人と何をしてきたかのLLM要約」を表示し、"人間関係の歴史" を振り返れるようにする。
- **なぜ**: entity + alias 機能が既にあるのが umi.mikan の隠れた強み。他の日記サービスにない差別化になる。
- **実装方針**:
  - alias 込みの言及検索を月別に集計する query を database 層に追加
  - 要約は既存の Pub/Sub 経路（`diary_events` チャネル）に新メッセージタイプを追加して非同期生成
- **ADR 0011 との関係**: 0011 は LLM による人物の**自動抽出**とグラフ可視化。本案はユーザが登録済みの entity を起点とした**時系列**の振り返りで、手動管理ベースのため先行実装しやすい。0011 実装時には抽出結果を entity に統合する設計が望ましい。

### 3. 「過去の自分に聞く」RAGチャット

**分類**: 振り返り→未来の橋渡し / **実装コスト**: 中〜大 / **LLM**: 必須

- **概要**: 「前に転職を迷ってたとき何を考えてた?」「最近疲れてるって書いたのいつ?」といった質問に、日記を根拠（引用元の日付リンク付き）で答える会話型インターフェース。
- **なぜ**: pgvector 埋め込みとセマンティック検索（ADR 0009）が既にあり、あと一歩でチャットにできる。コンセプトの中核になり得る機能。
- **実装方針**:
  - 既存の `SearchDiaryEntriesSemantic` で関連日記を取得し、コンテキストとして Gemini API に渡す RAG 構成
  - 回答には必ず引用元日記の日付を含め、該当日記へ遷移できるようにする（ハルシネーション対策）
  - 会話履歴の保持方針（セッション限り or DB永続化）は着手時に判断
  - `/llm` ページの RAG 図の更新が必要（CLAUDE.md の開発ワークフロー参照）

### 4. 目標・意図の抽出とフォローアップ

**分類**: 振り返り→未来の橋渡し / **実装コスト**: 中 / **LLM**: 必須

- **概要**: LLM が日記から「〜したい」「〜するつもり」という意図・目標を抽出して保存し、一定期間後に「これ、その後どうなりましたか?」と再提示する。ユーザは「達成した / まだ / やめた」を記録できる。
- **なぜ**: 「未来をよくする」方向で最も直接的に効く。振り返り（過去）と行動（未来）を橋渡しする、コンセプトそのものの機能。
- **実装方針**:
  - highlight 生成（ADR 0008）と同じ経路で、日記保存時に `diary_intent` メッセージタイプを追加して抽出
  - 新テーブル `diary_intents`（抽出元 diary_id、意図テキスト、状態、再提示予定日）
  - 再提示は Scheduler の `DailyScheduledJob` で実施。iOS はローカル通知にも載せる
  - 誤抽出があり得るため、ユーザが意図を却下・編集できる UI を必須とする

### 5. 感情・ムードの時系列可視化

**分類**: 振り返りの質向上 / **実装コスト**: 中 / **LLM**: 必須

- **概要**: 日記保存時に LLM で感情スコアを抽出し、月次・年次でグラフ化する。「毎年2月に落ち込む」「この人と会った週は気分がいい」といった本人も気づかないパターンを可視化する。
- **なぜ**: 単体でも価値があるが、エンティティ（2）やトレンド分析との掛け合わせ、年次レビュー（6）の素材として効く。
- **実装方針**:
  - Subscriber に `diary_emotion` メッセージタイプを追加し、日記ごとに感情スコア（数値 + 感情タグ）を抽出・保存
  - 新テーブル `diary_emotions`（diary_id、スコア、感情タグJSONB）
  - 過去日記のバックフィルは `RegenerateAllEmbeddings` と同様のオンデマンド一括処理方式
- **ADR 0010 との関係**: 0010 は期間指定の**レポート内**で感情トレンドを扱う（emotion_timeline は週単位・レポート生成時のみ）。本案は**日記1件ごと**に感情を永続化し、任意粒度でのグラフ描画・他機能との結合を可能にする。0010 実装時は本案のデータを入力として使うとレポート生成コストも下がる。

### 6. 年次レビュー（Year in Review）

**分類**: 振り返り導線 / **実装コスト**: 中 / **LLM**: 必須

- **概要**: 年末に「今年の10大トピック」「よく登場した人」「感情の推移」を自動生成してまとめる。
- **なぜ**: 月次サマリーの仕組みの自然な拡張。年に一度の「人生の棚卸し」体験を作る。
- **実装方針**:
  - 月次サマリー12ヶ月分を入力にして年次サマリーを生成（日記全文を送るよりトークン効率が良い）
  - Scheduler の `DailyScheduledJob` 基盤で12月末〜1月初に自動生成
  - 感情データ（5）とエンティティ集計（2）が貯まってから作ると効果が跳ねる

## 優先順位の考え方

```
1 → 2: 既存資産の再利用だけでほぼ作れて、体験の変化が大きい（先行実装）
3 → 4: コンセプトの本丸だが設計が重い（中核として腰を据えて実装）
5:      4・6 の素材になるので中盤に仕込む
6:      5 のデータが貯まってから（年末リリースを狙う）
```

## 優先度を下げる判断をしたもの

- **写真添付**: ストレージ・iOSオフライン同期の複雑さが跳ね上がる割に、「振り返って未来をよくする」への寄与が間接的
- **ソーシャル機能（共有・公開）**: 「自分の人生と向き合う」というプロダクトの性格と相性が悪い

## 結果

### メリット

1. **コンセプトの実現**: 「記録・検索」止まりだった体験が「振り返り→行動変容」まで届く
2. **既存アーキテクチャの活用**: Scheduler・Pub/Sub・pgvector・entity・WidgetKit をすべて再利用でき、新規インフラが不要
3. **段階的に価値を出せる**: 1・2 は LLM 依存が薄く小さく出せる。3〜6 は互いに素材を供給し合う

### デメリット・リスク

1. **LLM 依存機能の精度**: 意図抽出（4）・感情抽出（5）は誤りがあり得るため、ユーザによる修正手段が必須
2. **機能過多のリスク**: 日記の本質は「書くこと」。分析機能が主張しすぎると書く体験を損なうため、導線は控えめに保つ
3. **Proposed ADR との調整**: 0010・0011 と重なる領域があり、実装順によってはデータ設計の統合が必要（各節に記載）
